**Problem:** The builder implemented the operator's visual engagement criteria using the same divergence number as the already-ratified two-phase composition plan. The AMENDMENT-014 file says "Ratifies: DIVERGENCE-020" — which now ambiguously refers to two different documents. The CHANGELOG entry for DIVERGENCE-020 refers to the operator visual engagement criteria, not the composition plan amendment.

**Resolution:** Rename the operator visual engagement divergence to **DIVERGENCE-021**. Update the file name, the CHANGELOG entry, the PROGRESS.md entry, and the DIVERGENCE-020 reference in the validator comment at `edit_planning.py:1362`. AMENDMENT-014's "Ratifies: DIVERGENCE-020" correctly refers to the composition plan divergence (filed first).

### Finding 1.2: DIVERGENCE-021 (operator visual engagement) — APPROVED WITH CONDITIONS [STRATEGIC/LOGIC]

**Rationale:** The operator's direction is sound and within their authority. The 4-second visual change floor, caption emphasis, supporting visuals, VO-only visual life, video-over-stills preference, richer motion vocabulary, and scene coherence are all taste directives that correctly belong in prompts and validators, not code classes. The implementation places them in prompt templates (Writer, Visual Director, Edit Plan) and a mechanical validator — charter-compliant.

**Conditions:**

1. **Move `MAX_CLIP_DURATION` to config.** The hardcoded `MAX_CLIP_DURATION = 4.0` at `src/services/edit_planning.py:1347` is a business value in code. It must live in `config/render_styles.yaml` or `config/models.yaml` as `max_segment_seconds` (which already exists at lines 250-254 as variant-type-specific values but is ALSO hardcoded — see Finding 4.1). The validator should read from config, not a Python constant.

2. **Move `max_segment_seconds` variant-type mapping to config.** Lines 250-254 hardcode `max_segment_seconds = 3` for reel/short/carousel and `4` for other formats, using string matching on `variant_type`. This is judgment in code (format-specific pacing rules belong in config or the Format Guide, not Python if/elif chains).

3. **Prompt-level directives only.** The caption emphasis, supporting visuals, VO-only visual life, video-over-stills, motion vocabulary, and scene coherence criteria are correctly placed in prompt templates. Confirm no Python heuristics were added for these — they should be LLM judgment, not keyword matching.

4. **File as DIVERGENCE-021** (per Finding 1.1).

## Part 2 — M15 implementation review

### Finding 2.1: Service-layer architecture is charter-compliant [APPROVED]

The 14 new service files are architecturally sound:

- **No tenant strings** in any M15 service file (verified via grep for `stackpenni`, `StackPenni`, `sou-sou`, `Caribbean`, `island_futurist`).
- **No hardcoded brand colors or fonts** in service files (Montserrat/Anton in `composition_plan.py:210-212` are config fallback defaults, acceptable).
- **Immutable candidates** (`candidate_store.py`): append-only decisions, status lifecycle, no `approved` Boolean as source of truth — matches AMENDMENT-013.
- **Manifest freeze** (`manifest_freeze.py`): fail-closed on stale/rejected/superseded/rights-invalid/cost-unapproved, idempotent identical freeze — matches charter.
- **ProductionSession** (`production_orchestrator.py`): compare-and-set transitions, full AMENDMENT-013+014 state machine, transition history, durable states — matches charter.
- **RendererSpec** (`renderer_spec.py`): provider-neutral, capability registry, fail-closed lowering — matches DIVERGENCE-019.
- **Ratification** (`ratification_service.py`): spec-hash binding, stale detection, no false greens — matches AMENDMENT-014.

### Finding 2.2: Composition route does not use CompositionPlanGenerator [P0 — INTEGRATION GAP]

**File:** `src/app.py:9771-9806` (composition_ratification route)

**What:** The route builds a minimal empty plan inline:
```python
plan = {
    "schema_version": "1.0",
    "manifest_hash": manifest_data.get("manifest_hash", ""),
    "canvas": {"width": 1080, "height": 1920, "fps": 30.0, "aspect_ratio": "9:16"},
    "text_elements": [],
    "audio_elements": [],
    "visual_elements": [],
    "graphics_elements": [],
    "transitions": [],
}
```

It does NOT import or call `CompositionPlanGenerator` (built in VF-CP-001) or `CompositionPreviewGenerator` (built in VF-CP-002). The `generate()` method that compiles text/audio/visual/graphics/transition elements from the manifest and Writer contract is never invoked.

**Impact:** The composition ratification surface shows the operator an empty plan — no text roles, no audio lanes, no visual clips, no graphics, no transitions, no previews. The operator sees only a status badge and a plan hash. This makes the entire VF-CP-001..004 work functionally invisible to the operator.

**Charter breach:** AMENDMENT-014 clause: "The CompositionPlan declares every element of the final video" and "Per-element previews are generated locally before ratification." The route satisfies neither.

**Required fix:** Wire the route to use `CompositionPlanGenerator.generate()` and `CompositionPreviewGenerator` (or equivalent). The plan must contain real elements traced to the manifest. Previews must be generated. The hardcoded canvas (1080×1920, 30fps, "9:16") must come from the format/config, not inline Python.

### Finding 2.3: Component Workbench and Composition routes are orphaned [P0 — INTEGRATION GAP]

**What:** No page in the UI links to `/workbench/<asset_id>` or `/composition/<asset_id>`. Verified:
- Asset page (`/create/assets/22`): 0 links to `/workbench/`
- Assemble page (`/assemble`): 0 links to `/workbench/`
- Home page (`/`): 0 links to `/workbench/`
- Ideas page (`/ideas`): 0 links to `/workbench/`

**Impact:** The operator cannot discover or navigate to the Component Workbench or Composition Ratification surfaces. The M15 work is functionally invisible. The operator's only path is typing the URL directly.

**Required fix:** Add a "Component Workbench" link/button on the asset page (`/create/assets/<id>`) that appears when the asset is in a pre-Gate-3 state. Add a "Composition Plan" link that appears after manifest freeze. The assemble page should also link to the workbench for each asset.

### Finding 2.4: Workbench back-link points to wrong asset [P1 — BUG]

**File:** `src/app.py` (workbench route, template render)

**What:** The workbench for asset 22 shows: `← Back to asset 22 (Instagram)` but the href is `/create/assets/26` — a different asset. The back link uses the draft_id instead of the asset_id.

**Fix:** The back link should go to `/create/assets/{asset_id}` (the asset the workbench is for), not `/create/assets/{draft_id}`.

### Finding 2.5: Composition page — stale plan shows enabled Ratify button [P1 — STATE DISSONANCE]

**What:** On `/composition/20`, the page shows:
- Status badge: "Stale"
- Warning: "⚠️ This plan has changed since ratification. The previous ratification is invalid."
- But the "Ratify composition" button is **enabled**.

A stale plan should require re-review before ratification. The operator should see the updated plan elements and previews, not ratify a stale plan blindly. At minimum, the Ratify button should be disabled until the operator acknowledges the staleness.

### Finding 2.6: Composition page shows raw plan hash and ISO timestamp [P2 — JARGON/TIMESTAMPS]

**What:** The composition page displays:
- `Plan hash: f2bc39c361477110aacf54a70ed09ddaea603741c2bb6572e8de8661f872251d` — developer-facing
- `Previous ratification: ratify at 2026-07-23T10:49:48.393138+00:00` — raw ISO timestamp + "ratify at" is not English

**Fix:** Use a short hash display (first 8 chars) with a tooltip for the full hash. Use relative time ("ratified 6 hours ago"). Replace "ratify at" with "ratified".

## Part 3 — Deep 10-dimension UI walkthrough

### Dimension 1: State Dissonance

**Finding 3.1 [P1]:** Workbench shows "Planning components…" badge alongside "✅ All required components approved. Ready to freeze." — contradictory. If all components are approved, the session should be in `component_review_required` or `manifest_ready`, not `planning_components`. The badge should reflect the actual state.

**Finding 3.2 [P1]:** Workbench shows "Generate Component Requirements" button even when all components are already approved and registered. This button should not appear if requirements already exist and candidates are registered.

**Finding 3.3 [P1]:** Composition page shows "Stale" status but Ratify button is enabled (see Finding 2.5).

### Dimension 2: Staleness

**Finding 3.4 [P2]:** The composition page correctly detects staleness (plan hash mismatch) and shows a warning. This is good. However, it doesn't show what changed — the operator can't compare the old plan with the new one.

### Dimension 3: Technical Jargon

**Finding 3.5 [P2]:** Workbench page shows raw file paths to the operator: `📎 data/media/22/vo_take_1784808293.wav` and `📎 data/media/22/image_e996ded6178f85c8.png`. The operator should see "Voice recording (32 seconds)" or "Generated image (beat 1)", not filesystem paths with hash filenames.

**Finding 3.6 [P2]:** Composition page shows raw plan hash (see Finding 2.6).

**Finding 3.7 [P2]:** Asset page shows a raw ISO timestamp in the Inspiration source citation: `(Source: bundle_social_instagram_audio, collected 2026-07-20T12:53:44.922913+00:00)`. Should be relative time or friendly format.

**Finding 3.8 [P2]:** Asset page renders raw JSON dict content for Writer beats: `{'label': 'HOOK', 'vo_text': 'AI is giving...', 'text_on_screen': {...}, 'visual': {...}}`. This is developer-facing data, not operator-readable content. The beats should be formatted as readable cards, not raw Python dict repr.

### Dimension 4: Descriptive Titles

**Finding 3.9 [OK]:** Home page uses idea text as titles, not just IDs. "№ 061" alongside the actual idea text. Good.

**Finding 3.10 [P2]:** Workbench page title is just "Component Workbench" — doesn't say which piece. Should be "Component Workbench — [idea title] (Instagram Reel)".

**Finding 3.11 [P2]:** Composition page title is "Composition Ratification" — doesn't say which piece. Should include the asset/draft context.

### Dimension 5: Grouping

### Dimension 6: Timestamps

**Finding 3.13 [P2]:** Composition page shows raw ISO timestamp (see Finding 2.6).

**Finding 3.14 [P2]:** Asset page shows raw ISO timestamp in source citation (see Finding 3.7).

**Finding 3.15 [OK]:** Assemble page uses relative time ("4 hours ago", "11 hours ago"). Good.

### Dimension 7: Content Drift

**Finding 3.16 [OK]:** No evidence of fan-out content drift in the current assets. The Writer beats shown on the asset page appear to be the approved content.

### Dimension 8: Empty States

**Finding 3.17 [P2]:** Composition page with no manifest shows "No frozen manifest — freeze components first" as plain text (HTTP 400). This should be a proper page with navigation, not a bare error string. The operator gets no UI context.

**Finding 3.18 [OK]:** Published page shows a proper empty state: "Nothing published yet. Approve assets at Gate 3, then schedule them at Gate 4 to see published content here." Good.

### Dimension 9: Version Visibility

**Finding 3.19 [OK]:** Workbench shows candidate versions ("v1" badge). Good.

**Finding 3.20 [P2]:** Workbench candidate version is shown as small grey text (`font-size: 0.8rem; color: #757575`). The charter says version should be a visible badge, not tiny grey text.

### Dimension 10: False Greens

**Finding 3.21 [OK]:** Published page shows "0" for published/scheduled counts — no false greens. Buffer is not configured (empty API key) and the page doesn't claim connection.

**Finding 3.22 [P1]:** Workbench shows "✅ All required components approved. Ready to freeze." but the session state is `planning_components`. This is a false green — the system says "ready to freeze" but the state machine hasn't transitioned to `manifest_ready`. The readiness summary should match the session state.

**Finding 3.23 [P1]:** Workbench shows the "Freeze choices and assemble →" button enabled, but clicking it may fail because the session is in `planning_components` state, not `component_review_required` or `manifest_ready`. The freeze action should be gated on the correct state.

**File:** `src/services/edit_planning.py:249-254`
**What:**
```python
if "reel" in variant_type or "short" in variant_type:
```
